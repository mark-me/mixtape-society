M3U is a plain text playlist format that music apps understand. Example:


## Server Architecture

### Three New Endpoints

#### 1. `/download/<slug>/playlist.m3u`
- Generates M3U playlist file
- Contains URLs pointing to individual tracks
- Tiny file size (~2KB)
- Opens directly in music apps

#### 2. `/download/<slug>/track/<path>`
- Serves individual tracks
- Converts FLAC â†’ MP3 on-the-fly
- Supports HTTP range requests (for seeking)
- Cacheable responses

#### 3. `/download/<slug>/package`
- Original ZIP package (for desktop)
- All tracks + offline player + metadata
- Best for archiving

### Performance Considerations

**Caching Strategy**:
- Converted MP3s are **not** cached on disk (design choice)
- But marked as `Cache-Control: public, max-age=31536000`
- CDN or reverse proxy (nginx) can cache them
- Reduces server load for popular mixtapes

**Why not cache converted files?**
1. Avoid disk space issues (could be 100s of GB)
2. Conversion is fast enough (~10-30s per track)
3. Most users only play tracks once
4. Can implement disk caching later if needed

## Troubleshooting

### "M3U won't open"
**Cause**: No music app installed  
**Solution**: Instructions in modal + user guide

### "Tracks won't play"
**Cause**: Network issues or server error  
**Solution**: Check server logs, add retry logic

### "Conversion is slow"
**Cause**: Large FLAC files  
**Solution**: Increase FFmpeg threads: `-threads 4`

### "High server load"
**Cause**: Many concurrent conversions  
**Solution**: 
1. Implement conversion queue (Celery/RQ)
2. Add Redis caching for popular tracks
3. Use CDN with long cache times
